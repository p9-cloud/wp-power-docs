# Clarify Session 2026-04-20 17:31

## Idea

SortablePosts 排序 bug 修復需求。

使用者在知識庫管理後台（`#/docs/edit/:id` 的 SortablePosts tab）拖拽章節排序後，
POST `/posts/sort`（body: `{ from_tree, to_tree }`）回應成功，但部分情境下 DB 的
`menu_order` 或 `post_parent` 似乎沒有如預期更新，或更新後前端重新載入看到的樹狀結構
與拖拽結果不一致。

根因範圍可能橫跨兩層：
- power-docs 本地（`js/src/pages/admin/Docs/Edit/tabs/SortablePosts/index.tsx`
  的 `from_tree` / `to_tree` 組裝、lodash `isEqual` 比對、invalidate 時機）
- Powerhouse 共用層的 `CRUD::sort_posts`（儲存邏輯本身可能有問題）

Dev 階段第一步：先跑 SQL SELECT 確認資料庫 `wp_posts.menu_order` 與 `post_parent`
是否真的被寫入，再判斷根因位於哪一層。

Feature spec 需同時涵蓋兩種根因可能，待 DB 驗證後才能收斂。

## Q&A

- Q1: bug 根因範圍是只在 power-docs，還是可能包含 Powerhouse 共用層？
  → A: 不限定只在 power-docs，視為「影響範圍待查證，包含 Powerhouse 共用層可能有 bug」。
- Q2: 若最終根因在 Powerhouse 共用層，採 local override 還是直接修共用層？
  → A: 條件式——根因確認是 Powerhouse 問題 → 直接修 Powerhouse `CRUD::sort_posts`；
    根因僅在 power-docs → 走 local override。
- Q3: 先查 DB 還是先改碼？
  → A: C——dev 階段第一步跑 `SELECT` 確認 SQL 寫入與否，Feature spec 寫成
    「涵蓋兩種可能，根因待確認」。
- Q4: 測試情境涵蓋範圍？
  → A: E——同層級重排（A）＋跨層級移動（B）兩者皆驗。
- Q5: 持久化驗證斷言點？
  → A: C——驗 `wp_posts.menu_order` + `post_parent` + `post_modified` 三欄。
- Q6: Feature 檔案結構？
  → A: B——一個 Feature、兩個 Rule（對外行為 + 資料持久化），加 CiC(AMB)
    便條紙註記「根因層級待 SQL 驗證後決定」。
